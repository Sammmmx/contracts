# Staking

## What is the Staking Program?

The Staking program allows users to deposit (stake) SPL tokens into a pool and earn reward tokens over time. Rewards are distributed proportionally — the more tokens you stake relative to the total pool, the greater your share of the reward.

This is the same mechanism used by major DeFi protocols like Synthetix, Aave, and Compound. This implementation ports the EVM pattern to Solana using the Anchor framework.

# Core Mechanisms

## The Reward Accumulator Pattern

The program uses a "reward per token" accumulator to efficiently track how much each staker has earned without iterating over all users. This is the key insight that makes the pattern efficient regardless of how many stakers exist.

### How It Works

Instead of updating every user's reward balance on every stake/withdraw, the program maintains a single global value: `reward_per_token_stored`. This value represents the total reward earned per staked token since the beginning of the reward period.

```
reward_per_token = reward_per_token_stored
    + (elapsed_time * reward_rate * 1e18) / total_staked
```

Each user stores a snapshot of `reward_per_token` at the time of their last interaction (`reward_per_token_paid`). The difference between the current value and their snapshot, multiplied by their stake, gives their pending reward:

```
earned = (user_balance * (reward_per_token - reward_per_token_paid)) / 1e18
    + rewards_earned
```

### Why 1e18 Scaling?

Neither Solidity nor Rust have native floating-point support suitable for financial math. If `reward_rate` is small relative to `total_staked`, integer division would truncate to zero. Multiplying by `1e18` (stored as `PRECISION`) before dividing preserves precision. The factor is divided out when calculating the actual reward amount.

### Why `u128`?

The `reward_per_token_stored` and intermediate math values use `u128` to prevent overflow. With `u64` token amounts scaled by `1e18`, the intermediate product can reach ~10^37, which exceeds the `u64` max (~1.8×10^19) but fits comfortably in `u128` (~3.4×10^38).

## EVM vs. Solana: Key Differences

### Account Model vs. Storage Mappings

In EVM, the contract stores user balances in `mapping(address => uint256)` — all data lives inside the contract's storage. On Solana, each user's stake data lives in a separate **Program Derived Address (PDA)** account:

```rust
#[account]
pub struct UserStake {
    pub pool: Pubkey,
    pub owner: Pubkey,
    pub balance: u64,
    pub reward_per_token_paid: u128,
    pub rewards_earned: u64,
    pub bump: u8,
}
```

This PDA is derived from `[b"user_stake", pool_key, user_key]`. The user pays rent for this account on their first stake (via `init_if_needed`), and it persists across transactions.

### PDA Vaults vs. Contract Balance

In EVM, staked tokens are held "by the contract" — the contract address owns them. On Solana, tokens are held in **Associated Token Accounts (ATAs)** owned by the pool PDA:

```
pool PDA (authority) → staking vault ATA (holds staked tokens)
                     → reward vault ATA (holds reward tokens)
```

The pool PDA signs CPI (Cross-Program Invocation) calls to the Token program to transfer tokens out of the vaults. No single keypair controls the vaults.

### The `updateReward` Modifier → Helper Function

In Solidity, `updateReward` is a modifier that runs before the function body:

```solidity
modifier updateReward(address account) {
    rewardPerTokenStored = rewardPerToken();
    // ...
    _;
}
```

In Rust/Anchor, there are no modifiers. Instead, `update_reward` is a plain function called at the top of each instruction handler:

```rust
pub fn handler(ctx: Context<Stake>, amount: u64) -> Result<()> {
    update_reward(pool, Some(user_stake), &clock)?;
    // ... rest of the logic
}
```

The effect is identical — reward state is checkpointed before any balances change.

### Time Source

EVM uses `block.timestamp`. Solana uses the `Clock` sysvar, which provides `unix_timestamp`. Both represent seconds since epoch, so the reward math is the same. On Solana, `Clock::get()` is a syscall — no account needs to be passed.

### Reentrancy

EVM contracts need `ReentrancyGuard` because external calls (like token transfers) can call back into the contract. Solana's runtime processes each instruction atomically and prevents reentrancy by design — a program cannot be invoked by itself during CPI execution.

## The Reward Lifecycle

### 1. Initialization

The authority calls `initialize`, which creates:

- A **PoolState** PDA seeded by `[b"pool", staking_mint, reward_mint]`
- A **staking vault** ATA (the pool PDA's associated token account for the staking mint)
- A **reward vault** ATA (the pool PDA's associated token account for the reward mint)

The program enforces `staking_mint != reward_mint` to prevent vault accounting issues.

### 2. Funding

The authority transfers reward tokens directly to the reward vault (it's just a token account, anyone can send tokens to it). Then the authority calls `configure_reward(amount, duration)`, which sets:

- `reward_rate` — tokens distributed per second (`amount / duration`)
- `period_finish` — when the reward period ends (`current_timestamp + duration`)

The program verifies the vault holds enough tokens before accepting the configuration.

### 3. Accrual

Every time a user stakes, withdraws, claims, or exits, `update_reward` runs first:

```rust
pub fn update_reward(
    pool: &mut PoolState,
    user_stake: Option<&mut UserStake>,
    clock: &Clock,
) -> Result<()> {
    pool.reward_per_token_stored = reward_per_token(pool, clock)?;
    pool.last_update_time = last_time_reward_applicable(pool, clock);

    if let Some(stake) = user_stake {
        stake.rewards_earned = earned(pool, stake)?;
        stake.reward_per_token_paid = pool.reward_per_token_stored;
    }

    Ok(())
}
```

This checkpoints the global state and the user's individual reward balance before any balances change.

### 4. Claiming

When a user calls `claim_reward`, the function transfers accrued rewards from the reward vault to the user's reward token account. The pool PDA signs this CPI transfer using its seeds:

```rust
let seeds = &[b"pool", pool.staking_mint.as_ref(), pool.reward_mint.as_ref(), &[pool.bump]];
token::transfer(
    CpiContext::new_with_signer(token_program_id, transfer_accounts, &[seeds]),
    reward_amount,
)?;
```

The `rewards_earned` field resets to zero after the transfer.

## Token Flow

```
User → stake(amount) → staking tokens transfer to staking vault
User → withdraw(amount) → staking tokens transfer back from vault (PDA-signed)
User → claim_reward() → reward tokens transfer from reward vault (PDA-signed)
User → exit() → both of the above in one instruction
```

Transfers **into** vaults use the user as the signer (standard CPI). Transfers **out of** vaults use the pool PDA as the signer (CPI with signer seeds).

## The `exit()` Instruction

`exit()` is a convenience instruction that withdraws the user's full stake and claims all pending rewards in a single transaction. It inlines both operations directly rather than calling `withdraw` and `claim_reward` as separate CPIs — this is more compute-efficient and matches the EVM implementation.

# Security Considerations

## Checked Arithmetic

All math uses Rust's `checked_add`, `checked_sub`, `checked_mul`, and `checked_div` methods. If any operation overflows, the program returns a `MathOverflow` error instead of panicking or wrapping. This is Solana's equivalent of Solidity's built-in overflow protection (since 0.8.0).

## Reward Period Enforcement

The authority cannot start a new reward period while one is still active (`RewardPeriodNotFinished`). This prevents reward dilution — if the authority could restart mid-period, existing stakers would lose unclaimed rewards from the interrupted period.

## Balance Verification

`configure_reward` checks that the reward vault balance covers the announced reward amount. Without this check, the authority could promise rewards the vault can't deliver, causing later claims to fail.

## PDA Vault Security

Token vaults are ATAs owned by the pool PDA. No human keypair can directly withdraw from these accounts. Only the program, using the correct PDA seeds, can sign transfers. This eliminates custodial risk.

## Account Validation

Anchor's constraint system validates all accounts before the instruction handler runs:
- PDA seeds and bumps are verified
- Token account ownership and mint are checked
- The signer constraint ensures only the authorized user or authority can act

## Integer Division Dust

`reward_rate = reward_amount / duration` truncates. For a 7-day period, up to 604,799 units of the smallest denomination may remain undistributed. This is economically negligible.

## `init_if_needed` Security

The `UserStake` account uses `init_if_needed`, which creates the account on the user's first stake. The program checks `pool == Pubkey::default()` to detect first-time initialization and sets the correct fields. This feature requires the `init-if-needed` Cargo feature flag.

# Deployment

```bash
anchor build
anchor deploy --provider.cluster devnet
```

After deployment:

1. Call `initialize` with the staking and reward mint addresses
2. Transfer reward tokens to the reward vault (the pool PDA's ATA for the reward mint)
3. Call `configure_reward(total_reward_amount, duration_in_seconds)` from the authority

# Testing Coverage

The test suite validates the complete lifecycle of the staking program using the Solana test validator.

## Testing

```bash
anchor test --validator legacy
```

## 1. Initialization

Tests verify pool PDA creation with correct authority, mints, and vaults. Validates zero initial state and rejection of same staking/reward mint.

## 2. Staking

Tests cover single-user staking, multi-user staking, multiple stakes from the same user, token transfer verification, and zero-amount rejection.

## 3. Withdrawal

Tests cover full withdrawal, partial withdrawal, token return verification, zero-amount rejection, and insufficient balance rejection.

## 4. Reward Configuration

Tests cover authority-only access, duration validation, insufficient balance rejection, and prevention of overlapping reward periods.

## 5. Reward Accrual

Tests verify time-based accrual, proportional distribution between multiple stakers, full distribution at period end, accrual stops after the period finishes, and zero-staked behavior.

## 6. Reward Claiming

Tests cover token transfer to user, reward reset after claim, and no-op when nothing is earned.

## 7. Exit

Tests verify combined withdraw + claim behavior and revert when nothing is staked.

## 8. Re-configuration

Tests verify that a new reward period can be configured after the previous one ends.
