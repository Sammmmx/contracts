# ERC-20 Token Template

## What is ERC-20?

ERC-20 is the standard interface for fungible tokens on Ethereum and EVM-compatible blockchains. "Fungible" means every token is identical and interchangeable — 1 PHT is always equal to any other 1 PHT, just like how $1 is always equal to any other $1.

The standard defines a common set of functions that every ERC-20 token must implement, which allows wallets, exchanges, and other contracts to interact with any token without needing custom code for each one.

## Standard Functions

- `transfer(to, amount)` — send tokens to another address
- `approve(spender, amount)` — allow another address to spend your tokens
- `transferFrom(from, to, amount)` — transfer tokens on behalf of another address
- `balanceOf(address)` — check how many tokens an address holds
- `totalSupply()` — check total tokens in circulation
- `allowance(owner, spender)` — check how much a spender is allowed to spend

## This Template

This contract extends the standard ERC-20 with three additional features:

### Mintable

The owner can create new tokens up to a maximum supply cap:

```solidity
function mint(address to, uint256 amount) public onlyOwner {
    uint256 remaining = maxSupply - totalSupply();
    if (amount > remaining) revert ExceedsMaxSupply(amount, remaining);
    _mint(to, amount);
    emit Mint(to, amount);
}
```

The `maxSupply` is set at deployment and is immutable — it can never be changed after deployment.

### Burnable

Any token holder can permanently destroy their own tokens:

```solidity
token.burn(amount);           // burn your own tokens
token.burnFrom(from, amount); // burn someone else's tokens with their approval
```

Burning reduces `totalSupply` permanently.

### Pausable

The owner can pause all token movements in an emergency:

```solidity
token.pause();   // blocks all transfers, mints, and burns
token.unpause(); // resumes normal operation
```

Useful when a vulnerability is discovered and you need to stop all activity immediately.

## Mint Authority

Minting is restricted to the contract owner via OpenZeppelin's `Ownable`. This means:

- Only the address that deployed the contract can mint
- Ownership can be transferred to another address after deployment
- Ownership can be renounced entirely to make the token fully decentralized with a fixed supply

## Decimals

ERC-20 tokens use 18 decimal places by default. This means 1 token is stored as `1000000000000000000` (10^18) internally. So when minting 100 tokens you actually pass `100 * 10^18`.

## ERC-20 Permit

This template includes `ERC20Permit` which allows gasless approvals. Normally approving a spender requires a transaction that costs gas. With Permit, users can sign an approval off-chain and let someone else submit it on-chain, paying the gas for them. This is used heavily in DeFi protocols.

## Security Considerations

**Max Supply Cap**
Always set a max supply at deployment. Without a cap, a compromised owner key could mint unlimited tokens and destroy the token's value.

**Centralization Risk**
The owner has significant power — they can mint tokens and pause the contract. In production, document clearly who controls the owner key and consider transferring ownership after the initial mint is complete.

**Pausable Risk**
Pausable is a double-edged sword. It protects users in emergencies but also means the owner can freeze everyone's funds. Users should be aware of this when trusting a token.

**Reentrancy**
OpenZeppelin's ERC-20 implementation is safe from reentrancy attacks. The `_update` function updates state before any external interactions.

**Integer Overflow**
Solidity 0.8.0+ has built-in overflow protection. No SafeMath needed.

**Approval Front-Running**
The standard `approve` function has a known front-running issue. If you want to change an allowance from 100 to 50, an attacker can front-run and spend the 100 before the new approval goes through. Use `increaseAllowance` and `decreaseAllowance` to avoid this.

## Deployment

From the `evm/token-erc20` directory:

```bash
npx hardhat ignition deploy ignition/modules/ERC20deploy.ts --network sepolia
```

To deploy with custom parameters, put values in a JSON file (module id matches `buildModule` in `ignition/modules/ERC20deploy.ts`, here `TokenERC20Module`):

```json
{
  "TokenERC20Module": {
    "name": "Phantom",
    "symbol": "PHT",
    "maxSupply": "1000000000000000000000000"
  }
}
```

Then run:

```bash
npx hardhat ignition deploy ignition/modules/ERC20deploy.ts --network sepolia --parameters ./deploy-params.json
```

Constructor parameters:

- `initialOwner` — address that will own the contract and have mint authority. (This parameter is already set in ERC20deploy.ts)
- `name` — token name (e.g. "MyToken")
- `symbol` — token symbol (e.g. "MTK")
- `maxSupply` — maximum tokens that can ever exist (e.g. `1000000 * 10^18` for 1 million tokens)
