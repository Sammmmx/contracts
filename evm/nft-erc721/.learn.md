# NFT721

## What is the NFT721 Contract?

The NFT721 contract is a smart contract implementation based on the **ERC-721** standard for Non-Fungible Tokens. Unlike standard tokens where every unit is identical, each token in this contract is unique and identified by a discrete `tokenId`.

This contract serves as a feature-complete foundation for digital assets. It integrates multiple standards to handle verifiable ownership, on-chain token discovery, flexible metadata storage, and automated royalty enforcement, ensuring full compatibility with the broader NFT ecosystem and marketplaces.

# Core Mechanisms

## The Minting Lifecycle

The contract ensures each asset is unique through a controlled creation process that links ownership, metadata, and royalties in a single transaction.

### 1. Sequential ID Management

The contract uses a private `_nextTokenId` counter to manage asset identification. Every time a new token is created, the contract assigns the current counter value as the `tokenId` and then increments it. This guarantees that no two NFTs share the same ID and provides a clear audit trail of the collection's growth.

### 2. The safeMint Workflow

The `safeMint` function is restricted to the contract owner and handles the complete initialization of an NFT:

- **Validation**: Reverts with `NoURI()` if the metadata link is empty.
- **Creation**: Triggers `_safeMint` to assign the ID to the recipient.
- **Metadata**: Calls `_setTokenURI` to map the token to its specific IPFS/Arweave link.
- **Royalties**: If a receiver is provided, it sets a custom royalty for that specific token.

# Inheritance & Overrides

Because the contract inherits from multiple OpenZeppelin extensions, several functions must be manually overridden to resolve "diamond inheritance" conflicts in Solidity.

- **\_update**: This is the most critical override. It ensures that when a token moves, both the base `ERC721` balance logic and the `ERC721Enumerable` indexing logic are updated simultaneously.
- **\_increaseBalance**: Required to bridge the gap between the base token implementation and the enumeration tracking system.

# Standards & Interfaces

The NFT721 contract is designed for maximum interoperability, implementing several industry standards that allow it to communicate seamlessly with marketplaces, wallets, and explorers.

## 1. ERC-165 Identification

To ensure compatibility, the contract uses **ERC-165** to signal which features it supports. When an external platform (like OpenSea) interacts with the contract, it calls `supportsInterface` to "ask" if the contract can handle specific standards like Royalties or Enumeration.

# Royalty Enforcement (ERC-2981)

The contract implements a standardized way to handle secondary market royalties, ensuring creators are compensated whenever their work is resold.

## 1. The Basis Point System

Royalties are calculated using **Basis Points (bps)**, where $100\%$ is represented as $10,000$. This allows for high precision (e.g., $2.5\%$ is $250$ bps).

- **Maximum Cap**: The `checkFee` modifier prevents fees from exceeding $100\%$, protecting the mathematical integrity of the calculations.

## 2. Global vs. Local State

The contract supports two layers of royalty management:

- **Default Royalty**: Established in the `constructor` or updated via `setDefaultRoyalty`. This applies to every token in the collection.
- **Token-Specific Royalty**: Set during the `safeMint` process. This allows specific assets to have unique royalty percentages or different receivers than the rest of the collection.

# Advanced Storage & Discovery

The NFT721 contract provides a high level of transparency and data flexibility by enabling on-chain indexing and individual metadata management.

## 1. Enumeration Logic

Through **ERC721Enumerable**, the contract maintains a searchable record of all tokens. This enables two critical features:

- **Global Discovery**: Users can iterate through every token minted using `tokenByIndex`.
- **Owner Discovery**: Wallets can instantly list all tokens owned by a specific address using `tokenOfOwnerByIndex`.

This removes the need for centralized databases to show a user's collection, as the data is served directly from the blockchain.

## 2. URI Flexibility

By utilizing **ERC721URIStorage**, the contract moves away from a "fixed" collection style.

- **Unique Metadata**: Every `tokenId` can point to a completely different CID or URL.
- **Data Integrity**: The `tokenURI` function is overridden to ensure it retrieves the specific string assigned during the `safeMint` process, ensuring the asset's attributes and image remain linked to the correct token.

# Security

## Custom Errors

Instead of revert strings, the contract uses custom errors which are cheaper to deploy and execute:

- `NullAddress()` — thrown when `address(0)` is provided for a royalty receiver
- `InvalidFee(uint256 _fee)` — thrown when a royalty fee exceeds 10,000 basis points
- `NoURI()` — thrown when an empty metadata URI is passed to `safeMint`

## Access Control

The contract inherits OpenZeppelin's `Ownable`, restricting critical functions to the contract owner only:

- `safeMint` — only the owner can create new tokens
- `setDefaultRoyalty` — only the owner can update royalty settings

Any unauthorized call to these functions is automatically reverted by the `onlyOwner` modifier.

## Fee Cap Enforcement

The `checkFee` modifier runs before any function that sets a royalty fee, ensuring the value never exceeds 10,000 basis points (100%). This prevents mathematically broken royalty configurations from being set either at deployment or during minting.

```solidity
modifier checkFee(uint96 _fee) {
    if (_fee > 10000) revert InvalidFee(_fee);
    _;
}
```

## Zero-Address Royalty Guard

If a `royaltyReceiver` of `address(0)` is passed to `safeMint` alongside a non-zero fee, the contract reverts with `InvalidFee`. This prevents royalty payments from being burned by accident.

## Safe Minting

The contract uses `_safeMint` instead of `_mint`. This triggers the `onERC721Received` hook when minting to a smart contract address, ensuring tokens cannot be permanently locked in contracts that don't support ERC-721.

## Sequential Token IDs

Token IDs are managed by a private `_nextTokenId` counter that is never exposed externally. This prevents ID manipulation or collision — every token is guaranteed a unique, predictable identifier.

```bash
npx hardhat ignition deploy NFT721Deploy.ts --network sepolia
```

```bash

```

# Testing Coverage

The `NFT721` contract is supported by a comprehensive test suite designed to validate the entire lifecycle of a token, from deployment constraints to edge-case security.

## Testing

```bash
npx hardhat test
```

## 1. Deployment & Initialization

Tests verify that the contract deploys with the correct parameters and handles invalid inputs gracefully:

- **Parameter Check**: Confirms the `name`, `symbol`, and `initialOwner` are set correctly.
- **Royalty Validation**: Ensures the constructor reverts if the `defaultFee` exceeds $10,000$ bps.
- **Access Control**: Validates that the deployer is granted the `Ownable` admin role.

## 2. Minting Logic

The test suite covers the "Happy Path" and mandatory requirements for creating assets:

- **Successful Mint**: Verifies that `safeMint` correctly assigns the `tokenId`, sets the `tokenURI`, and increments the internal counter.
- **Metadata Requirement**: Ensures the transaction reverts with `NoURI()` if an empty string is provided.
- **Royalty Mapping**: Confirms that both default and token-specific royalties are correctly reported by the `royaltyInfo` function.

## 3. Security & Access Control

Critical tests focus on preventing unauthorized access and protecting the state:

- **Owner-Only Restrictions**: Checks that non-owner addresses cannot call `safeMint` or `setDefaultRoyalty`.
- **Zero-Address Protection**: Validates that the contract reverts with `NullAddress()` when attempting to set a royalty receiver to `address(0)`.
- **Enumeration Integrity**: Verifies that `totalSupply` and `tokenOfOwnerByIndex` update accurately after every transfer or mint.

## 4. ERC-721 Standards Compliance

The suite includes technical checks for standard implementation:

- **Interface Support**: Confirms that `supportsInterface` returns `true` for all implemented standards (ERC-721, Enumerable, URIStorage, and ERC-2981).
- **Safe Transfers**: Ensures the contract correctly handles transfers to other smart contracts using the `onERC721Received` hook.
