# Staking

## What is the Staking Contract?

The Staking contract allows users to deposit (stake) ERC-20 tokens into a pool and earn reward tokens over time. Rewards are distributed proportionally — the more tokens you stake relative to the total pool, the greater your share of the reward.

This is the same mechanism used by major DeFi protocols like Synthetix, Aave, and Compound to incentivize liquidity providers and protocol participants.

# Core Mechanisms

## The Reward Accumulator Pattern

The contract uses a "reward per token" accumulator to efficiently track how much each staker has earned without iterating over all users. This is the key insight that makes the pattern gas-efficient regardless of how many stakers exist.

### How It Works

Instead of updating every user's reward balance on every stake/withdraw, the contract maintains a single global value: `rewardPerTokenStored`. This value represents the total reward earned per staked token since the beginning of the reward period.

```
rewardPerToken = rewardPerTokenStored
    + (elapsedTime * rewardRate * 1e18) / totalStaked
```

Each user stores a snapshot of `rewardPerToken` at the time of their last interaction (`userRewardPerTokenPaid`). The difference between the current `rewardPerToken` and their snapshot, multiplied by their stake, gives their pending reward:

```
earned = (userBalance * (rewardPerToken - userRewardPerTokenPaid)) / 1e18
    + rewards[user]
```

### Why 1e18 Scaling?

Solidity has no floating-point numbers. If `rewardRate` is small relative to `totalStaked`, the division would truncate to zero. Multiplying by `1e18` before dividing preserves precision. The `1e18` factor is divided out when calculating the actual reward amount.

## The Reward Lifecycle

### 1. Funding

The owner transfers reward tokens to the contract and calls `configureReward(amount, duration)`. This sets:

- `rewardRate` — tokens distributed per second (`amount / duration`)
- `periodFinish` — when the reward period ends (`block.timestamp + duration`)

The contract verifies it actually holds enough tokens before accepting the configuration. This prevents phantom rewards that can't be paid out.

### 2. Accrual

Every time a user stakes, withdraws, or claims, the `updateReward` modifier runs first:

```solidity
modifier updateReward(address account) {
    rewardPerTokenStored = rewardPerToken();
    lastUpdateTime = lastTimeRewardApplicable();
    if (account != address(0)) {
        rewards[account] = (balanceOf[account]
            * (rewardPerTokenStored - userRewardPerTokenPaid[account])) / 1e18
            + rewards[account];
        userRewardPerTokenPaid[account] = rewardPerTokenStored;
    }
    _;
}
```

This "checkpoints" the global state and the user's individual reward balance before any balances change.

### 3. Claiming

When a user calls `claimReward()`, the modifier updates their pending reward, and then the function transfers those tokens out. The `rewards[account]` mapping is reset to zero after the transfer.

# Staking and Withdrawal

## Token Flow

```
User → stake(amount) → stakingToken transfers to contract
User → withdraw(amount) → stakingToken transfers back to user
User → claimReward() → rewardToken transfers to user
User → exit() → both of the above in one call
```

The contract uses OpenZeppelin's `SafeERC20` for all token transfers. This handles tokens that don't return a boolean on `transfer`/`transferFrom` (a common non-standard behavior) and tokens that revert on failure.

## The exit() Function

`exit()` is a convenience function that withdraws the user's full stake and claims all pending rewards in a single transaction. It inlines the withdraw and claim logic directly to avoid extra external calls, while the `updateReward` modifier handles reward checkpointing.

# Security Considerations

## Reentrancy Protection

All user-facing functions (`stake`, `withdraw`, `claimReward`) use OpenZeppelin's `ReentrancyGuard`. This prevents a malicious token contract from re-entering the staking contract during a `safeTransfer` callback.

## Reward Period Enforcement

The owner cannot start a new reward period while one is still active (`RewardPeriodNotFinished`). This prevents reward dilution — if the owner could restart mid-period, existing stakers would lose unclaimed rewards from the interrupted period.

## Balance Verification

`configureReward` checks that the contract's reward token balance covers the announced reward amount. Without this check, the owner could promise rewards the contract can't deliver, causing later claims to revert.

## Integer Division Dust

`rewardRate = rewardAmount / duration` truncates. For a 7-day period, up to 604,799 wei of reward tokens may remain undistributed. This is economically negligible but important to understand: the contract will always have a tiny leftover balance.

## Same Token Risk

If `stakingToken == rewardToken`, the balance check in `configureReward` includes staked balances. This means the owner could configure rewards using staker deposits as the "funding." Always use separate tokens for staking and rewards.

## No Lock Period

Users can withdraw at any time with no penalty. This is intentional — it keeps the contract simple and composable. Protocols that need lock-up periods should extend this contract with a `lockDuration` per user.

## Access Control

Only the owner can call `configureReward`. All other functions are open to any address. The owner has no ability to withdraw user stakes or interfere with reward claims.

# Testing Coverage

The test suite validates the complete lifecycle of the staking contract using two mock ERC-20 tokens.

## Testing

```bash
npx hardhat test
```

## 1. Deployment & Constructor Validation

Tests verify that the contract initializes with correct token addresses and owner, and reverts on zero-address inputs.

## 2. Staking

Tests cover single-user staking, multi-user staking, multiple stakes from the same user, token transfer verification, and zero-amount rejection.

## 3. Withdrawal

Tests cover full withdrawal, partial withdrawal, token return verification, zero-amount rejection, and withdrawal without a prior stake.

## 4. Reward Configuration

Tests cover owner-only access, duration validation, insufficient balance rejection, and prevention of overlapping reward periods.

## 5. Reward Accrual

Tests verify time-based accrual, proportional distribution between multiple stakers, full distribution at period end, and accrual stops after the period finishes.

## 6. Reward Claiming

Tests cover token transfer to user, event emission, reward reset after claim, and no-op when nothing is earned.

## 7. Exit

Tests verify combined withdraw + claim behavior and revert when nothing is staked.

## 8. Ownership

Tests cover ownership transfer and unauthorized access rejection.
