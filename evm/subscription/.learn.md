# Subscription Contract

## What is a Subscription Contract?

A Subscription Contract is a smart contract that automates recurring payments between subscribers and merchants on the blockchain. Think of it like a decentralized version of a Netflix or Spotify subscription — except instead of a company charging your credit card, a smart contract handles the payment logic transparently and trustlessly.

Merchants define subscription plans with a name, price, and billing duration. Users subscribe to these plans and pay with an ERC20 token. Merchants can trigger auto-renewals when a billing period ends, and subscribers can pause, resume, or cancel at any time.

---

## Roles

### 1. Authority

- This role has an exclusive permission to Initiate the process of Ownership Transfer.
- Authority comes into action in case of Owner loses access to its address.
- This is an immutable role meaning once set cannot be transferred to a new address.

### 2. Owner

- Merchant Registrations, Revoking Registrations and Transfer of Ownership can only be handled by the Owner.
- Ownership can be transffered to an new address. Authority or Existing Owner must Initiate the Process.

### 3. Merchants

Mercants can Define Plans for their Subscriptions, trigger auto-renewal for subscribers after completing of billing period and withdraw full amount anytime.

### 4. Subscriber

Subscribers can purchase as many subscriptions additonally allowing them to pause, resume or unsubscribe without refund.

## Core Mechanisms

### 1. Merchant Registration

Before a merchant can create a plan, the contract owner must register them. This acts as a whitelist — only approved merchants can offer subscriptions on the platform.

```solidity
function registerMerchant(address _merchant) public onlyOwner
```

### 2. Transfer of Ownership

Ownership can be transferred to a new address following several reasons with the main highlight on the case of current owner losing keys or ownership rights. Only the current Owner and Authority can act as the Initiators of this process.

```solidity
function transferOwnership(address newOwner) public
```

### 3. Revoking a Merchant

Owner can remove a registered merchant stating their valid reasons. Merchant loses permission to create subscription plans and existing plans of the merchant are deactivated. Merchants can still complete their pending withdrawals from their subscriptions. Subscribers will have to manually cancelSubscription after billing period is over.

```solidity
function revokeMerchant(address _merchant public) onlyOwner
```

### 4. Defining a Plan

Once registered, a merchant can define one or more subscription plans. Each plan has a name, duration (in days), and a price (in tokens). Every plan gets a unique ID that subscribers use to sign up.

```solidity
function definePlan(string memory _name, uint256 _duration, uint256 _price) public
```

### 5. Subscribing

A user calls `subscribe` with the plan ID they want. The contract checks their token balance, transfers the first payment, and records the subscription details — including when the next billing date is.

```solidity
function subscribe(uint256 subscriptionID) public
```

> **Important:** Before subscribing, the user must approve the contract to spend their tokens using the ERC20 `approve` function.

### 6. Subscriptions Renewal

The merchant calls `subscriptionsRenewal` once the billing period has ended. The contract verifies the period is complete and charges the subscriber for the next cycle.

```solidity
function autoRenewal(address _subscriber, uint256 subscriptionID) public
```

### 7. Pause and Resume

Subscribers can pause their subscription at any time to stop auto renewals. When they resume, if the billing date has already passed, the contract charges them immediately for the missed cycle.

```solidity
function pauseRenewal(uint256 subscriptionID) public
function resumeRenewal(uint256 subscriptionID) public
```

### 8. Cancel

Subscribers can cancel their subscription entirely. This resets all their subscription data, and the merchant can no longer auto-renew them.

```solidity
function cancelSubscription(uint256 subscriptionID) public
```

### 9. Merchant Withdrawal

Payments from subscribers are held in the contract. Merchants call `merchantWithdrawal` to pull their accumulated earnings for a specific plan.

```solidity
function merchantWithdrawal(uint256 subscriptionID) public
```

### 10. Deactivate Plan

Merchants can deactivate a plan to stop new subscriptions from being created. Existing subscribers are not affected but auto renewals are also blocked on a deactivated plan.

```solidity
function deactivatePlan(uint256 _subscriptionID) public
```

## Assumptions

- **Single token payments** — the contract accepts only one ERC20 token set at deployment. There is no support for multiple payment tokens.
- **No refunds** — cancelling a subscription does not refund the current billing period. The subscriber keeps access until the period naturally ends but no funds are returned.
- **Fixed pricing** — once a plan is defined, its price and duration cannot be changed. If a merchant wants different terms, they must create a new plan.
- **Auto renewal is permissioned merchant triggered** - The contract does not automatically charge subscribers. The merchant or an off-chain bot must call `autoRenewal` when the billing period ends. **_Note_**: In case of offline merchants, the subscriptions silently freeze while still appearing subscribed in `Subscribers[...]`.
- **Token approval required** — subscribers must approve the contract to spend their tokens before subscribing. This is standard ERC20 behaviour but easy to miss.

## Security Considerations

### SafeERC20

The contract uses OpenZeppelin's `SafeERC20` wrapper instead of calling `transfer` and `transferFrom` directly. Some ERC20 tokens do not correctly return a boolean on these calls — `SafeERC20` handles this safely and reverts if the transfer fails.

### Reentrancy Guard

The contract inherits OpenZeppelin's `ReentrancyGuard`. This prevents an attacker from calling back into the contract during a token transfer to drain funds or manipulate state.

### Custom Errors

Instead of `require` strings, the contract uses custom errors like `error InvalidID()`. These are cheaper on gas and make it easier to identify exactly what went wrong in a failed transaction.

### Access Control

- Only the **owner** can register merchants
- Only the **merchant of a plan** can trigger auto renewals, withdrawals, and deactivations
- Only the **subscriber** can pause, resume, or cancel their own subscription

### Deactivated Plan Check

Deactivated plans block new subscriptions and auto renewals. However, existing subscribers can still cancel — they are never permanently locked in.

## Deployment

The contract takes two constructor parameters:

| Parameter | Description                                         |
| :-------- | :-------------------------------------------------- |
| `_owner`  | The address that will manage merchant registrations |
| `_token`  | The ERC20 token address used for payments           |

Deploy using Hardhat Ignition. Pass the token address at deploy time via a `parameters.json` file:

```json
{
  "SubscriptionModule": {
    "tokenAddress": "0xYourTokenAddressHere"
  }
}
```

Then run:

```bash
npx hardhat ignition deploy ignition/modules/SUBSCRIPTIONDeploy.ts --network sepolia "Parameters"
```

> Make sure your `.env` file has `ALCHEMY_URL`, `PRIVATE_KEY`, and `ETHERSCAN_API_KEY` set before deploying.

## Test Coverage

The test suite covers all core functions across multiple scenarios. A `TokenERC20` contract is deployed alongside the subscription contract in tests to simulate real ERC20 payments.

| Section                | What is Tested                                                                                                                                                        |
| :--------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **registerMerchant**   | Owner-only access, zero address, duplicate registration, successful registration, event emission                                                                      |
| **definePlan**         | Unregistered merchant, empty name, zero duration, zero price, successful plan creation, event emission                                                                |
| **subscribe**          | Deactivated plan, non-existent plan, duplicate subscription, insufficient balance, successful subscription, token deduction, billing date calculation, event emission |
| **autoRenewal**        | Deactivated plan, wrong merchant, no active subscription, paused subscription, period not complete, successful renewal, event emission                                |
| **merchantWithdrawal** | Wrong merchant, nothing to withdraw, successful transfer, balance reset                                                                                               |
| **deactivatePlan**     | Wrong merchant, already deactivated, successful deactivation, blocks new subscriptions, event emission                                                                |
| **pauseRenewal**       | No active subscription, already paused, successful pause, event emission                                                                                              |
| **resumeRenewal**      | Deactivated plan, no active subscription, not paused, successful resume, charge on overdue billing, no charge if not overdue, event emission                          |
| **cancelSubscription** | No active subscription, state fully reset, blocks auto renewal, allows resubscription, event emission                                                                 |
